```
  вҖў Observadores de Laravel escuchan por torrents nuevos o actualizados
```

**Por quГ© es importante**: Buscar en mГЎs de 50,000 torrents toma **milisegundos** en lugar de segundos. La base de datos se mantiene ligera. Los usuarios obtienen resultados instantГЎneos y filtrados.

### 2. **рҹ“§ Lista Negra de Correos Resiliente**

**El Problema**: UNIT3D obtiene dominios de correo desechables de un CDN externo durante la validaciГіn del registro. **Si el CDN estГЎ caГӯdo o inaccesible, los registros fallan por completo.**

**Nuestra SoluciГіn - Estrategia de Lista Negra HГӯbrida**:

```
PRIMARIO (Online):
  вң… Obtiene una lista actualizada del CDN (andreis/disposable-emails)
  вң… Actualiza una vez al arrancar (php artisan auto:email-blacklist-update)

RESPALDO (Offline):
  вң… Almacena una copia local: storage/app/email-blacklist.json
  вң… MГЎs de 7,160 dominios persistidos localmente
  вң… Si el CDN no es accesible, usa la cachГ© local (el registro sigue funcionando)

PERSISTENCIA:
  вң… La cachГ© sobrevive a los reinicios de los contenedores
  вң… La cachГ© sobrevive a `docker compose down/up`
  вң… La cachГ© se incluye en los backups completos
```

**Detalles de ImplementaciГіn**:
- Se creГі `app/Helpers/EmailBlacklistUpdater.php` (lГіgica de actualizaciГіn automГЎtica)
- El entrypoint ejecuta `php artisan auto:email-blacklist-update` al arrancar
- Un comando de Artisan personalizado vigila el CDN y escribe en el JSON local
- El registro usa la cachГ© local como principal (mГЎs rГЎpido, fiable)

**Resultado**: El registro funciona **incluso si el CDN estГЎ caГӯdo**. El sistema es autГіnomo y capaz de funcionar sin conexiГіn.

### 3. **рҹҢҗ Transparencia de Direcciones IP (Redes de Docker)**

**El Problema**: En Docker, Nginx y la aplicaciГіn Laravel se ejecutan en contenedores separados. Sin las cabeceras adecuadas, todas las peticiones parecen provenir de la puerta de enlace interna de Docker (`172.21.0.1`). **Todos los usuarios muestran la misma IP en sus perfiles.**

**Nuestra SoluciГіn - Cabeceras de Proxy Inverso + TrustProxies**:

```
CAPA DE NGINX (.docker/nginx/default.conf):
  вҖў proxy_set_header X-Real-IP $remote_addr;
  вҖў proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
  вҖў proxy_set_header X-Forwarded-Proto $scheme;

CAPA DE LARAVEL (app/Http/Middleware/TrustProxies.php):
  вҖў protected $proxies = '*';  [Confiar en Nginx como proxy inverso]
  вҖў Lee la cabecera X-Real-IP y la usa como la IP de origen del usuario

RESULTADO:
  вң… IPs reales de los usuarios capturadas en la base de datos
  вң… Cada usuario ve su IP pГәblica real en su perfil
  вң… El baneo y las estadГӯsticas basadas en IP funcionan correctamente
```

**VerificaciГіn**: Inicia sesiГіn, visita tu perfil вҶ’ verГЎs tu IP pГәblica real, no la puerta de enlace de Docker.

### 4. **рҹ”’ ProtecciГіn contra Fuerza Bruta: Equilibrio entre Seguridad y Usabilidad**

**El Problema**: La configuraciГіn por defecto de Fortify en UNIT3D era **demasiado agresiva**:
- 5 inicios de sesiГіn fallidos вҶ’ bloqueado por 24 horas
- IP de puerta de enlace compartida Гәnica (172.21.0.1 en Docker) вҶ’ usuarios legГӯtimos bloqueados juntos

**Nuestro Ajuste** (`app/Providers/FortifyServiceProvider.php`):

**Seguridad Adicional**:
- Se creГі la cuenta `BackupOwner` con permisos completos (acceso de emergencia)
- Se puede usar la cuenta de respaldo si la principal estГЎ bloqueada
- Los logs rastrean los intentos fallidos para investigar ataques reales

**Resultado**: El sistema protege contra la fuerza bruta **mientras permite la recuperaciГіn y pruebas legГӯtimas**.

#### **Backup en FrГӯo (Snapshot QuirГәrgico)**

**FilosofГӯa**: Los backups deben ser **a prueba de corrupciГіn, completos y capaces de funcionar sin conexiГіn**.

```bash

   в””в”Җ Se usa para la reconstrucciГіn sin conexiГіn si Docker Hub no estГЎ disponible

   в””в”Җ Comprime: cГіdigo de la aplicaciГіn, vendor/, node_modules/, configuraciones, datos
   в””в”Җ Incluye todo lo necesario para una recuperaciГіn completa sin conexiГіn

5. вҷ»пёҸ RotaciГіn (mantiene los Гәltimos 3 snapshots)
   в””в”Җ Evita que el disco se llene, mantiene backups recientes

6. рҹҡҖ ResurrecciГіn (docker compose up)
   в””в”Җ Verifica que el backup se haya realizado con Г©xito
   в””в”Җ Reinicia el sistema inmediatamente (minimiza el tiempo de inactividad)
```

**ВҝPor quГ© "quirГәrgico"?**:
- вң… **Sin corrupciГіn**: Detener los contenedores asegura la consistencia de los archivos durante la copia
- вң… **Plug & Play**: Se incluyen `vendor/` y `node_modules/` completos
- вң… **RecuperaciГіn sin conexiГіn**: ImГЎgenes de Docker + todas las dependencias = funciona en cualquier lugar
